四、定时任务模块设计

定时任务运行在 CHK 的 evolve-daemon 中，负责知识库的持续健康维护和数据同步。

健康检查
*/30 * * * *

每 30 分钟检查知识库健康状态

  • 检查孤立页面（无入链/出链）
  • 检测断链和失效链接
  • 验证 index.json 一致性
  • 检查矛盾内容（与进化系统联动）
  • 报告异常到 effect_tracking.jsonl
双向同步
*/15 * * * *

每 15 分钟同步 CHK 和 llm-wiki 数据

  • CHK web_captured → llm-wiki raw/
  • llm-wiki 新页面 → CHK 索引
  • 同步成熟度/置信度状态
  • 更新 cross-reference
  • 处理删除的级联清理
效果评估
0 2 * * *

每天凌晨 2 点评估知识有效性

  • 统计 usage_count 变化
  • 计算知识引用率
  • 晋升达到阈值的条目
  • 降级长期未使用的知识
  • 生成效果报告
内容更新
0 6 * * *

每天早上 6 点检查网页内容变化

  • 检测已抓取页面的变化
  • 更新摘要和嵌入向量
  • 标记过时内容
  • 重新运行 digest（如有必要）
  • 通知用户重要更新
Lint 检查
0 3 * * 0

每周日凌晨 3 点深度清理

  • 全面检查矛盾页面
  • 清理 orphan pages
  • 补充缺失的 cross-reference
  • 优化 index.md 结构
  • 生成 lint 报告
进化分析
0 4 * * *

每天早上 4 点运行进化分析

  • 分析效果追踪数据
  • 生成候选本能记录
  • 评估知识库增长趋势
  • 识别高频/低频领域
  • 更新进化历史

五、定时任务架构

⏱
调度策略

分层调度：高频任务（15-30min）× 低频任务（Daily/Weekly）

智能错峰：避免同时执行多个重任务

失败重试：指数退避，最多重试 3 次

并发控制：同时最多 2 个任务执行

📊
执行日志

scheduler.log：记录每次执行的开始/结束/结果

effect_tracking.jsonl：写入效果评估数据

sync_state.json：记录同步状态，支持断点续传

告警机制：连续失败 3 次触发通知

六、实现时间线

Phase 1: 最小集成
预计 1-2 天
创建 web_captured/ 目录 → 扩展 knowledge_recommender.py → 新增 Hook collect_web.py → 集成 llm-wiki ingest 命令
Phase 2: 双向同步
预计 2-3 天
实现 CHK ↔ llm-wiki 同步脚本 → 统一数据格式 → 实现统一检索接口
Phase 3: 定时任务
预计 2 天
实现 scheduler.py → 健康检查任务 → 双向同步任务 → 效果评估任务
Phase 4: 深度融合
预计 3-5 天
会话结晶化 → 图谱集成 → 进化系统联动 → Lint 检查任务 → 内容更新任务
