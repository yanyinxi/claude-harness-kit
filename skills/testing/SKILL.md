# 测试技能

## 何时使用

当需要以下操作时使用此技能：
- 设计测试用例
- 编写单元测试
- 执行集成测试
- 生成测试报告

## 工作流程

### 第一步：分析需求
- 阅读验收标准
- 识别测试场景
- 列出测试点

### 第二步：设计测试用例
- 正向测试（Happy Path）
- 负向测试（边界情况）
- 异常处理测试

### 第三步：编写测试代码
- 使用合适的测试框架
- 遵循测试金字塔原则
- 保持测试独立

### 第四步：执行测试
- 运行单元测试
- 运行集成测试
- 记录测试结果

**MCP 工具优先**：如果 E2E 测试涉及浏览器交互，优先使用 MCP 工具：
- `playwright` — 浏览器自动化测试、截图验证、DOM 操作
- 检查方式：查看工具列表是否有 `mcp__playwright` 前缀的工具
- 如已安装：直接用 playwright 做 E2E 验证，比写脚本更快更准
- 如未安装：提示用户 `claude mcp add playwright -- npx -y @playwright/mcp`

### 第五步：生成报告
- 汇总测试结果
- 记录失败的测试
- 提供修复建议

## 输出格式

```markdown
# 测试报告：[功能名称]

## 测试概览
- 总测试数：XX
- 通过：XX
- 失败：XX
- 通过率：XX%

```

## 最佳实践

1. **测试金字塔**：大量单元测试，适量集成测试，少量 E2E 测试
2. **独立性**：每个测试独立运行
3. **可重复**：测试在任何环境可重复执行
4. **快速反馈**：测试应该快速执行
5. **测试位置**：所有测试必须放在 main/tests/ 目录下

## 测试场景分类模板

### 1. 正向测试（Happy Path）
- 核心功能正常工作
- 标准数据处理
- 正常业务流程

### 2. 逆向测试（Negative Path）
- 错误输入处理
- 异常数据检测
- 边界情况处理

### 3. 边界测试（Edge Case）
- 最小值/最大值
- 空值/null 处理
- 临界状态

### 4. 异常测试（Exception Handling）
- 文件缺失降级
- 网络异常恢复
- 超时处理
